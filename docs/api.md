# API — контракт фронт ↔ бэк

Контракт мобильного клиента и Go-бэкенда: источник правды по эндпоинтам, токенам и обработке ошибок. Продуктовые детали флоу — в [auth-flow.md](./auth-flow.md).

## Соглашения

- Базовый путь: `/api/v1`.
- Формат — `application/json` (кроме загрузки фото).
- Авторизация — заголовок `Authorization: Bearer <accessToken>`. **Все эндпоинты требуют Bearer, кроме `/auth/*`** (в т.ч. все справочники/НСИ).
- Идентификаторы — `UUID`.
- Справочники (НСИ) — единый тип `{ id, value }`: `id` (`UUID`) — для ссылок из прикладных сущностей; `value` — стабильный ключ (напр. `WINE`), человекочитаемый перевод — на фронте (i18n по ключу). Ссылаемся **по `id`**.
- Время — по смыслу типа: **момент** → ISO 8601 / RFC3339 в UTC (`2026-06-21T10:30:00Z`); **календарная дата** (напр. `birthDate`) → `YYYY-MM-DD` без таймзоны; **длительность** (напр. `expiresIn`) → целое число секунд. Не используем unix-timestamp (ловушка секунды/мс). На клиенте в state/DTO держим время **строкой**, `Date` — только локально для показа (см. примечания к клиенту).
- Ошибки — единый конверт; `code` машиночитаемый, `message` для логов/дебага (не для прямого показа юзеру), `details?` — пофилдовые ошибки валидации. В шорткате ниже (`← 422 CODE { ... }`) `CODE` = `error.code`.
    ```json
    { "error": { "code": "VALIDATION_ERROR", "message": "...", "details": { "<field>": "<msg>" } } }
    ```
- **Тело запроса проверяется строго:** неизвестные/лишние поля → `422 VALIDATION_ERROR` (`details.body`). Слать ровно описанные поля. Email регистронезависим (нормализуется на бэке).
- **Коды ошибок, которые отдаёт сервер:**
    - auth: `VALIDATION_ERROR`(422), `RU_EMAIL_REQUIRED`(422), `RATE_LIMITED`(429), `OTP_INVALID`(401), `OTP_EXPIRED`(401), `SESSION_REVOKED`(401), `INVALID_REFRESH`(401), `UNAUTHORIZED`(401);
    - профиль: `PROFILE_NOT_FOUND`(404), `STEP_ORDER`(409), `VALIDATION_ERROR`(422), `PROFILE_INCOMPLETE`(422), `PHOTO_NOT_FOUND`(404);
    - общее: `INTERNAL`(500).

## API

### Авторизация

Passwordless: **вход = регистрация**, один путь email + OTP. Новизна юзера — по флагу `isCreatedProfile`, а не по факту входа.

#### `POST /auth/send_code`

Единая точка отправки кода: **первый запрос, «отправить снова» и «сменить почту» — все дёргают этот метод.** Каждый вызов выпускает новый код и **инвалидирует предыдущий** (валиден только последний). Ответ одинаков, есть аккаунт или нет (не палим существование).

```
→ { "email": "user@example.com" }
← 200 { "expiresIn": 300, "resendAfter": 60 }
```

- `expiresIn` — сколько живёт код (сек);
- `resendAfter` — кулдаун до повторной отправки на **тот же** email (сек); раньше — `429 RATE_LIMITED`;
- rate-limit keyed by email: «сменить почту» = новый адрес = свой кулдаун, кулдауном прежнего не блокируется.

**Региональное правило (РФ).** Если IP клиента по GeoIP = `RU`, почта должна быть
российской (`.ru`/`.su`/`.рф`/punycode `.xn--p1ai`), иначе `422 RU_EMAIL_REQUIRED`
(`details.email`). Для не-RU и при выключенном GeoIP ограничения нет — блокировок
по стране нет, это мягкое правило (VPN-обход допустим). В `ENV=dev` заголовок
`X-Debug-Country: RU` переопределяет GeoIP для тестов.

Ошибки: `422 VALIDATION_ERROR`, `422 RU_EMAIL_REQUIRED`, `429 RATE_LIMITED`.

#### `POST /auth/check_code`

Проверяет код. При успехе: создаёт юзера (если нет), **ревокает все прежние сессии** (single-session), заводит новую, отдаёт токены.

```
→ { "email": "user@example.com", "code": "123456" }
← 200 { "accessToken": "<jwt>", "refreshToken": "<opaque>", "isCreatedProfile": false, "onboardingStep": "BASICS" }
```

Ошибки: `401 OTP_INVALID`, `401 OTP_EXPIRED`, `429 RATE_LIMITED`.

#### `POST /auth/refresh`

Ротация пары токенов. Если сессию перебили входом с другого устройства — `401 SESSION_REVOKED`.

```
→ { "refreshToken": "<opaque>" }
← 200 { "accessToken": "<jwt>", "refreshToken": "<opaque>", "isCreatedProfile": false, "onboardingStep": "ABOUT" }
```

Ошибки: `401 SESSION_REVOKED`, `401 INVALID_REFRESH`.

#### `POST /auth/logout`

Голый запрос с Bearer, без тела. Ревокает текущую (единственную) сессию.

```
Authorization: Bearer <accessToken>
← 204
```

#### `GET /me`

Идентичность аккаунта.

```
Authorization: Bearer <accessToken>
← 200 { "id": "<uuid>", "email": "...", "isCreatedProfile": false, "onboardingStep": "ABOUT" }
```

### Профиль

Создаётся в онбординге, **по шагам — у каждого шага свой метод с серверной валидацией.** Клиентской валидации не доверяем — дублируем на сервере. Шаг сохраняется как upsert (идемпотентно; частичный прогресс переживает выход из приложения). Имена — по смыслу (`basics` / `about`), не по номеру шага. Поля и правила — [auth-flow.md](./auth-flow.md). Все эндпоинты — Bearer.

**Шаги последовательные:** сервер не пускает дальше, пока предыдущий не пройден валидно (`409 STEP_ORDER`). Прогресс отдаётся полем `onboardingStep` ∈ `BASICS` → `ABOUT` → `PHOTOS` → `DONE` — это экран, который фронт показывает при возврате в онбординг. Поэтому `isCreatedProfile` одного мало: пока он `false`, экран берём из `onboardingStep`. Инвариант: `isCreatedProfile === (onboardingStep === "DONE")`.

#### `GET /me/profile`

```
← 200 <Profile> | 404 PROFILE_NOT_FOUND
```

#### `PUT /me/profile/basics` — шаг 1

```
→ {
    "name": "...",
    "gender": "MALE",                    // MALE | FEMALE
    "birthDate": "1998-05-20",
    "city": "..."
  }
← 200 { "profile": <Profile>, "onboardingStep": "ABOUT" }
```

Валидация: `name` обязателен; `gender` обязателен, ∈ enum (`MALE` | `FEMALE`); `birthDate` обязателен и **18+**; `city` обязателен.

#### `PUT /me/profile/about` — шаг 2

```
→ {
    "interestIds": ["<uuid>"],          // 1–5, должны быть в справочнике
    "description": "...",                 // 30–1000 символов, обязательно
    "datingGoal": "DATES",               // FRIENDSHIP | LONG_TERM_RELATIONSHIP | DATES | NOTHING_SERIOUS
    "height": 180,                        // см, опц., 100–250
    "weight": 72,                         // кг, опц., 30–300
    "showZodiac": true
  }
← 200 { "profile": <Profile>, "onboardingStep": "PHOTOS" }
```

Валидация: `interestIds` **1–5** и существуют в справочнике; `datingGoal` ∈ enum; `description` **30–1000 символов, обязательно**; `height` **100–250 см**, `weight` **30–300 кг** (оба опциональны, `float`). Знак зодиака вычисляется на бэке из `birthDate` (юзер вводом не управляет, только тоггл показа); **в `Profile` его нет** — фронт выводит из `birthDate` при `showZodiac=true`. Рост/вес — СИ в БД (см. [units.md](./units.md)).

#### Фото — шаг 3

Объект `Photo`:

```
{
  "id": "<uuid>",
  "url": "https://.../photo.jpg",
  "position": 0,                                 // 0 = главное (аватар), дальше по порядку
  "crop": { "x": 0.1, "y": 0.05, "size": 0.8 }   // нормализованный (0..1) квадрат под круглый аватар; только у главного, иначе null
}
```

Главное фото — `position: 0`, у него есть `crop` (область под круглый аватар). Остальные грузятся без кропа. «Сделать главным» = переупорядочить + задать новому главному `crop`; кроп меняется и отдельно. `crop` — нормализованные координаты в `[0,1]` (квадрат, в который вписан круг), независимы от размера картинки.

```
POST   /me/profile/photos        — multipart/form-data, поле "photo" → Photo (201)
PATCH  /me/profile/photos/{id}   — { "crop": { "x", "y", "size" } }  — задать/изменить кроп → Photo (200)
PATCH  /me/profile/photos/order  — { "order": ["<id>"] }             — порядок; первый = главное → Photo[] (200)
DELETE /me/profile/photos/{id}   — 204
```

Поток: грузим фото (`POST`), для главного на экране кропа шлём `PATCH {id}` с `crop`. Свап главного — `PATCH order` + `PATCH {id}` с новым кропом.

- **Транспорт загрузки — `multipart/form-data`, поле файла `photo`** (один файл за запрос). Ответ — объект `Photo` со `201`.
- **Формат/размер:** `image/jpeg`, `image/png`, `image/webp`; **≤ 10 МБ**. Иначе `422 VALIDATION_ERROR`.
- **Потолок 5** фото — при загрузке (`422`, если больше), **минимум 2** — на `complete`.
- **`PATCH order` принимает ПОЛНЫЙ список** id текущих фото ровно один раз (не частичный) — иначе `422`. Reorder сбрасывает `crop` у всех неглавных; у главного (`position 0`) `crop` сохраняется.
- Загрузка/правка фото доступны только начиная с шага `PHOTOS` (после `about`), иначе `409 STEP_ORDER`. Несуществующее фото → `404 PHOTO_NOT_FOUND`.

#### `POST /me/profile/complete`

Финальный гейт: проверяет, что все шаги заполнены валидно + **≥2 фото**, и атомарно ставит `isCreatedProfile = true`.

```
← 200 { "isCreatedProfile": true, "onboardingStep": "DONE" }
← 422 PROFILE_INCOMPLETE { "error": { "details": { "photos": "min 2" } } }
```

Ошибки шагов — `422 VALIDATION_ERROR { "error": { "details": { "<field>": "<msg>" } } }`. Порядок шагов **последовательный**: на следующий нельзя без валидного предыдущего (`409 STEP_ORDER`); финал — `complete`.

### Справочники

#### `GET /interests`

Справочник интересов — массив справочного типа `{ id, value }` (НСИ, см. «Соглашения»). Список управляется админкой (набор фронт не хардкодит); `value` — ключ, перевод на фронте по i18n. Неизвестный ключ (свежий, без перевода) — фронт показывает фолбэк.

```
← 200 [ { "id": "<uuid>", "value": "WINE" } ]
```

### Сущности (этого контракта)

- **User** (auth): `id`, `email`, `isCreatedProfile`, `createdAt`.
- **Profile** (1:1 с User): `id`, `userId`, `name`, `gender` (`MALE` | `FEMALE`), `birthDate`, `city`, `interests[]` (`{ id, value }`), `description`, `datingGoal`, `height?`, `weight?`, `showZodiac`, `photos[]` (`Photo[]`).
- **Photo**: `id`, `url`, `position`, `crop?` (см. «Фото»).

Пример `<Profile>`:

```json
{
  "id": "<uuid>",
  "userId": "<uuid>",
  "name": "Аня",
  "gender": "FEMALE",
  "birthDate": "1998-05-20",
  "city": "Москва",
  "interests": [{ "id": "<uuid>", "value": "COFFEE" }],
  "description": "...",
  "datingGoal": "DATES",
  "height": 168,
  "weight": null,
  "showZodiac": true,
  "photos": [
    { "id": "<uuid>", "url": "https://.../1.jpg", "position": 0, "crop": { "x": 0.1, "y": 0.05, "size": 0.8 } },
    { "id": "<uuid>", "url": "https://.../2.jpg", "position": 1, "crop": null }
  ]

```

Сущности встреч (`MeetingAd`, `MeetingCandidate`, …) и **фильтры поиска** («кого ищу»: пол/цель/возраст и т.п. — это критерии дискавери, **не** поля профиля) — в [matching.md](./matching.md); их раздел контракта добавим при работе над модулем встреч.

## Примечания к бэку

> **Статус реализации:** auth и профиль/онбординг (включая фото) реализованы как описано ниже. Живой источник схем — Swagger (`/swagger/index.html`). Значения, бывшие «TBD», зафиксированы по тексту выше.

- **Токены.** access — JWT HS256, TTL **15 мин**, claims только `{ sub, iat, exp }`. refresh — непрозрачный, хранится **хэшем** в БД, TTL **60 дней**, **ротируется** на каждом `refresh`.
- **`isCreatedProfile` не в JWT** — флаг меняется (false→true после онбординга), протухнет. Отдаём его в ответах `check_code` / `refresh` и в `GET /me`.
- **Single-session.** На `check_code` ревокаем все прежние refresh юзера и заводим один новый. Старый refresh → `401 SESSION_REVOKED`.
- **«Один вход» — eventual, не мгновенный.** Старый access (stateless JWT) живёт до своего TTL — на каждый запрос в БД не ходим (ради простоты MVP). Поэтому TTL access держим коротким; мгновенный кик старого access (stateful-проверка через кэш на каждый запрос) сознательно **не** делаем.
- **OTP** — 6 цифр; TTL **300с** (`expiresIn`), кулдаун **60с** (`resendAfter`); до **5 попыток** ввода на код, затем код инвалидируется. В dev-режиме код всегда **`000000`** (письма не шлются).
- **Rate-limit `send_code` — два независимых слоя:**
    1. **per-email** — кулдаун (`resendAfter`) + лимит отправок на один адрес; `429` по одной почте **не** блокирует другие;
    2. **per-client (IP / устройство)** — отдельный лимит на общее число отправок **по разным почтам** за окно. Без него «сменить почту» обходит per-email кулдаун → спам кодами по чужим ящикам и слив бюджета на письма.
- **Знак зодиака** вычисляется из `birthDate`; в `Profile` не отдаётся (фронт выводит сам).
- **Хранилище фото** — загрузка **multipart через бэкенд** (поле `photo`), за интерфейсом `Storage` с выбором провайдера по ENV: локальная ФС (dev) / S3-совместимое — MinIO/AWS S3/R2 (prod). URL в `Photo.url` — абсолютный, на базе публичного адреса стораджа.
- **Настройки приложения** (`unitSystem` / `theme` / `notifications`) — хранятся **локально на устройстве** (SQLite), на бэк не синкаются; в контракте их нет.

## Примечания к клиенту

- **Хранение токенов.** access — в памяти; refresh — в `expo-secure-store`.
- **Bootstrap (`initAuth`)** — на старте всегда `refresh`:
    ```
    нет refresh   → не авторизован → логин (без сети)
    есть refresh  → refresh()
                      ok  → берём isCreatedProfile + onboardingStep → роутинг (main / онбординг на нужном шаге)
                      401 → чистим токены → логин
    ```
    `refresh` возвращает `isCreatedProfile` + `onboardingStep`, поэтому `/me` на старте не нужен — один запрос на буте.
- **Интерсептор (единая точка по 401).** `401` — всегда сигнал по сессии, обрабатывается здесь, до саги:
    - `401` на любом запросе → `refresh` → **повтор исходного запроса один раз** → саге уходит реальный ответ (про 401 она не знает);
    - `refresh` упал → диспатч `logout` → экран входа;
    - сам запрос `refresh` **не** проходит через 401-ретрай (нет петли); повтор исходного — максимум один раз.
- **Классы ошибок.** `401` — про сессию, **никогда не показываем как «техническую»** (её гасит интерсептор). `5xx`/сеть — техническая. `4xx`-бизнес (валидация и т.п.) — показываем по месту. В саге: `if (isAuthError) skip; else show(...)`.
- **Single-flight refresh.** Параллельные `401` ждут **один общий** промис `refresh`, затем все повторяются — иначе ротация передерётся и сам себе словишь `401`.
- **Оффлайн на старте.** Сетевой сбой `refresh` ≠ `401`: токены не чистим, не разлогиниваем, повторяем позже.
