# Анкета встречи (`meeting-ad`) и лента (`meeting-list`)

Контракт анкеты (`MeetingAd`), ленты, лайков и чата. Продуктовая логика (роли,
отклики, чаты) — в [matching.md](./matching.md); соглашения (пути, токены, конверт
ошибок, типы, НСИ) — в [api.md](./api.md); локация/город/дистанция — в
[geolocation.md](./geolocation.md).

Статус: **контракт** (auth/профиль реализованы; локация/город — редизайн, см. geolocation.md).

## Сущность `MeetingAd`

```json
{
  "id": "<uuid>",
  "authorProfileId": "<uuid>",
  "description": "Иду на кофе в центр, ищу компанию",
  "tags": [{ "id": "<uuid>", "value": "COFFEE" }],
  "city": { "id": "<uuid>", "name": "Санкт-Петербург" },
  "status": "ACTIVE",
  "createdAt": "2026-06-22T10:30:00Z",
  "expiresAt": "2026-06-23T10:30:00Z"
}
```

- `tags` — НСИ `{ id, value }` из [meeting-tags.md](./meeting-tags.md), ссылка по `id`.
- `city` — **эффективный город автора** (для показа), берётся из автора, на анкете не
  хранится. Координаты не отдаём (приватность). `meeting_ads` координат/`city_id` не
  хранит — дистанция и город считаются из локации автора (см. geolocation.md). В ленте
  добавляется `distanceKm`. Времени встречи нет.
- `status` ∈ `ACTIVE` | `EXPIRED` | `CLOSED`.

## Справочник тегов

### `GET /meeting-tags`
```
← 200 [ { "id": "<uuid>", "value": "COFFEE" } ]
```

## Анкета (`meeting-ad`) — все Bearer

| Метод | Путь | Назначение |
|-------|------|------------|
| POST | `/meeting-ad` | создать активную анкету → `201 MeetingAd` |
| GET | `/meeting-ad/me` | моя активная (или `404 NO_ACTIVE_AD`) |
| GET | `/meeting-ad/{id}` | любая анкета по id (инфо не приватная) |
| PUT | `/meeting-ad/me` | отредактировать мою активную |
| DELETE | `/meeting-ad/me` | снять мою активную (→ `CLOSED`) |

### `POST /meeting-ad`
```
→ { "description": "...",        // 30–500, обязательно
    "tagIds": ["<uuid>"],        // 1–3, из справочника, без дублей
    "cityId": "<uuid>" }         // опц.: плашка «Поменять» → переопределить город
← 201 <MeetingAd>
← 403 PROFILE_REQUIRED           // профиль не завершён (isCreatedProfile=false)
← 409 ACTIVE_AD_EXISTS           // уже есть активная (одна на юзера)
← 422 VALIDATION_ERROR
```
Если `cityId` передан → переключает локацию **профиля** в `MANUAL` на этот город
(запоминается). Анкета координат не хранит — город/дистанция берутся из эффективной
локации автора. См. [geolocation.md](./geolocation.md).

### `PUT /meeting-ad/me`
Редактируются **только** `description` и `tagIds`. `createdAt`/`expiresAt`/`status`
не меняются (правка не продлевает жизнь). Полученные лайки сохраняются.
```
→ { "description": "...", "tagIds": ["<uuid>"] }
← 200 <MeetingAd> | 404 NO_ACTIVE_AD | 422 VALIDATION_ERROR
```

### `GET /meeting-ad/{id}` / `GET /meeting-ad/me`
```
← 200 <MeetingAd> | 404 (NO_ACTIVE_AD для /me; иначе AD_NOT_FOUND)
```
`{id}` доступен любому авторизованному (анкету видно при переходе на профиль).

### `DELETE /meeting-ad/me`
```
← 204 | 404 NO_ACTIVE_AD
```

## Лента (`meeting-list`) — discovery

### `GET /meeting-list`
Список чужих анкет рядом. Фильтры/сортировка/курсор — в query; клиент
**виртуализирует** список (бесконечная подгрузка по `nextCursor`, не offset).

```
GET /meeting-list?sort=distance&radius=25&gender=FEMALE
    &ageMin=18&ageMax=99&goal=DATES&tagIds=<id>&tagIds=<id>&cursor=<opaque>&limit=20
← 200 {
    "items": [ { /* MeetingAd + краткий профиль автора + distanceKm|null */ } ],
    "nextCursor": "<opaque>"   // отсутствует/null = конец
  }
```

- **Два режима поиска (взаимоисключающие):**
    - **`radius`** (float, км) — анкеты в N км от моей эффективной точки. **Нужна моя
      локация**, иначе пустая лента.
    - **`cityId`** — все анкеты, чей **автор в этом городе** (эффективный город автора =
      `cityId`), радиус не ограничиваем. Работает **даже без моего гео** (на случай
      сломанного GPS); `distanceKm` тогда `null`.
- **`sort`**: `distance` (ближайшие первыми) | `date` (новые первыми). По умолчанию `distance`.
- `distanceKm` — целые км (`0`=«<1 км»), либо `null` (city-режим без моей точки).
- **Курсор keyset** кодирует ключ сортировки + id; привязан к выбранному `sort`
  (сменил сортировку/режим → `cursor` пустой, лента заново).
- **Исключаются:** своя анкета и анкеты, на которые уже откликался; только `ACTIVE`.

#### Курсор и стабильность ленты

- Курсор **непрозрачный**: клиент хранит последний `nextCursor` и шлёт обратно;
  `nextCursor: null` → конец. Курсор привязан к `sort` (distance-курсор в `date` → `422`).
  При смене `sort`/фильтров/радиуса — курсор сбрасывается (лента с нуля).
- **`sort=date` стабилен** — ключ (`createdAt`) неизменен, новые анкеты приходят сверху
  и не пересекают путь пагинации.
- **`sort=distance` — дрейф возможен** (дистанции меняются, пока люди двигаются).
  Поэтому:
    - **Бэк: пин origin.** Курсор несёт точку отсчёта зрителя (его **собственные**
      координаты), и все страницы сессии считаются от неё — движение самого зрителя
      ленту не сдвигает. Чужих координат в курсоре нет.
    - **Клиент: дедуп по `id`** при подгрузке страниц (выкидывать анкеты, чей `id`
      уже в списке) — убирает видимые дубли от любого дрейфа. Пропуск (редко) лечится
      pull-to-refresh.
- Realtime «появились новые анкеты рядом» (тост → refresh) — отдельный модуль
  (presence + гео-pub/sub + websocket), не часть этого контракта.

## Лайки (отклики) — Bearer

Поскольку активная анкета одна, лайки всегда привязаны к ней.

| Метод | Путь | Назначение |
|-------|------|------------|
| POST | `/meeting-ad/{id}/like` | лайкнуть чужую анкету (зритель) |
| GET | `/likes` | входящие лайки на мою активную (автор) |
| POST | `/likes/{id}/accept` | принять → создаётся чат |
| POST | `/likes/{id}/reject` | отклонить |

```
POST /meeting-ad/{id}/like
← 201 | 409 ALREADY_RESPONDED | 422 CANNOT_LIKE_OWN_AD | 404 AD_NOT_FOUND | 409 AD_NOT_ACTIVE

GET /likes        // входящие на мою активную; [] если активной анкеты нет
← 200 [ {
    "id": "<uuid>",                 // id лайка (кандидата)
    "createdAt": "...",
    "profile": { "id": "<uuid>", "name": "Аня", "avatarUrl": "https://.../1.jpg" }
  } ]

POST /likes/{id}/accept → 200 { "chatId": "<uuid>" }   // accepted + создаётся чат
POST /likes/{id}/reject → 204
```
В списке лайков — **компактно** (имя + аватар для карточки). Клик по аватару →
экран профиля, полный профиль грузится через **`GET /profiles/{id}`**. Исходящие
лайки в MVP не показываем (matching.md): лайкнул → приняли → чат в мессенджере.

### `GET /profiles/{id}`

Публичный просмотр профиля по id (инфо не приватная — как анкета). Bearer.
```
← 200 <Profile> | 404 PROFILE_NOT_FOUND
```

## Чат — минимально

Пока только **сущность**, создаётся при `accept`: ссылается на пару (профиль
собеседника выводится из принятого лайка). Сообщения, таймер, заморозка — отдельным
этапом ([matching.md](./matching.md)).

## Жизненный цикл

- **Одна активная на юзера.** `POST` при `ACTIVE` → `409 ACTIVE_AD_EXISTS`.
- **`ACTIVE`** → **`EXPIRED`** через **24 часа** (ленивое истечение: просроченная
  `ACTIVE` флипается в `EXPIRED` при операциях автора).
- **`CLOSED`** — согласован постоянный чат (чат-модуль) или ручной `DELETE`.
- Пересоздание — клиент хранит последнюю анкету локально и шлёт новый `POST`.

## Ошибки

`PROFILE_REQUIRED`(403), `ACTIVE_AD_EXISTS`(409), `NO_ACTIVE_AD`(404),
`AD_NOT_FOUND`(404), `AD_NOT_ACTIVE`(409), `CANNOT_LIKE_OWN_AD`(422),
`ALREADY_RESPONDED`(409), `VALIDATION_ERROR`(422).
