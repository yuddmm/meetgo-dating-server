# Геолокация и города

Как приложение работает с местоположением: GPS-координаты, города (газеттир из
geonames), переопределение города, приватность, дистанция. Соглашения (пути,
токены, конверт ошибок, типы) — см. [api.md](./api.md). Лента/фильтры — в
[meeting-list.md](./meeting-list.md).

> Статус: **контракт/дизайн.** Частично меняет уже реализованное (город в
> онбординге, лента по «живому» гео) — будет отрефакторено при внедрении.

## Принципы

- **Координаты наружу не отдаём — никогда.** Иначе любой с токеном вытащит адрес.
  Клиент **присылает** координаты; дистанцию **считает бэкенд** и отдаёт готовой.
- **Город = производное от GPS + переопределяемое и запоминаемое.** Город НЕ вводят
  руками в онбординге: показываем по факту (reverse-geocode по GPS), но пользователь
  может выбрать вручную, и выбор запоминается.
- **PostGIS** — спатиал-индекс (GiST) + KNN-оператор `<->`: быстрый «ближайший город»
  и поиск по расстоянию. Расширение ставится в PostgreSQL (локально и в прод).

## Города (газеттир)

- Таблица `cities` из **geonames**: `id` (geonameId), `name` (международное/дефолтное),
  `name_ru` (локализованное из geonames `alternateNames`, isolang=ru), регион, страна,
  `location` (центроид, PostGIS `geography(Point)`), `population`.
- **Локализация:** показываем `name_ru`, если есть; иначе `name` (для зарубежных без
  русского имени — англ./транслит). MVP — **РФ** (фильтр country=RU), но **апдейтер
  пишем сразу под глобальный geonames**, чтобы масштабироваться без переделки.
- **Тулза-апдейтер:** скачивает дампы geonames (`cities*.txt` + `alternateNames`),
  парсит, делает upsert в `cities`. Запускается разово/периодически (данные стабильны).
- **Reverse-geocode** «координата → ближайший город» — PostGIS KNN по `cities.location`.

## Локация пользователя

### GPS — `POST /my-geo`
Записывает текущую локацию (upsert, одна строка на юзера). Bearer.
```
→ { "lat": 55.751244, "lng": 37.618423, "accuracy": 35 }   // accuracy — метры, опц.
← 204
```
- Клиент шлёт **при входе** + раз в **10 минут**. Координаты **не возвращаются** нигде.
- **Качество GPS (защита от лагов):**
    - игнорим мусорные фиксы: `accuracy` хуже порога (напр. > 5–10 км) → не обновляем;
    - **гард «неправдоподобный прыжок»**: точка ушла на >X км за <Y минут (нереальная
      скорость) → не обновляем (держим последнюю). Это и ловит «вдруг Кингисепп».
- Валидация: `lat∈[-90,90]`, `lng∈[-180,180]`.

### Режим локации и override
Профиль хранит `location_mode` (`AUTO` | `MANUAL`) и `manual_city_id`:
- **AUTO** (по умолчанию): эффективная локация = GPS-координата; эффективный город =
  ближайший к ней (reverse-geocode).
- **MANUAL**: эффективная локация = **центроид выбранного города** (фиксированная точка
  → идемпотентность дистанций); эффективный город = выбранный. Выбор **запоминается**.

### `PUT /me/location` — переключение
```
→ { "mode": "MANUAL", "cityId": "<uuid>" }   // MANUAL требует cityId
→ { "mode": "AUTO" }                          // вернуться к GPS (сбросить override)
← 200 { "mode": "...", "city": { "id", "name" } | null }
```

### `GET /cities?q=пит` — пикер города
Автокомплит для выбора города (список «Поменять местоположение»). Bearer.
```
← 200 [ { "id": "<uuid>", "name": "Санкт-Петербург", "region": "..." } ]
```

## Город в анкете (`MeetingAd`)

- **`meeting_ads` не меняется и координат не хранит.** Дистанция и город анкеты при
  выдаче берутся из **эффективной локации автора** (уровень юзера): `MANUAL` →
  центроид города, `AUTO` → GPS; город — выбранный/ближайший. Анкета просто отдаёт
  готовую `distanceKm` (или `null`) + `city`.
- При создании показываем плашку «Вы сейчас в `<город>`» (эффективный город автора) +
  «Поменять» → `GET /cities` → меняет `location_mode`/`manual_city` **на профиле**
  (запоминается). Анкета сразу отражает новый город.
- **Снимок локации на анкете НЕ делаем** — анкета «следует» за автором (сменил
  локацию → активная анкета показывает новую). Это удобно для фикса кривого GPS.

## Дистанция (в выдаче)

- Считает бэкенд; в ответе — **целые километры**: `0` = «<1 км» (фронт пишет «<1 км»),
  иначе **математическое округление** (`1.5 → 2`).
- Может быть `null` (см. city-режим ниже — когда у зрителя нет своей точки).

## Поиск (discovery) — два режима

Взаимоисключающие (либо то, либо то):
- **radius** — в пределах N км от моей эффективной точки (нужна моя локация).
- **city** — все анкеты, чей **автор в этом городе** (эффективный город автора =
  `cityId`); радиус не ограничиваем. Дистанцию показываем, если есть моя точка, иначе
  `distanceKm: null`. Город можно листать **даже при сломанном GPS** (дистанции null).

Полный контракт ленты/фильтров/курсора — в [meeting-list.md](./meeting-list.md).

## Приватность
- Сырые координаты не покидают бэкенд; наружу — только округлённая дистанция и город.
- Точное место встречи участники согласуют в чате.

## Хранение/масштаб
- На уровне юзера храним эффективную точку + `resolved_city_id` (для быстрых дистанции
  и city-фильтра). `meeting_ads` локацию НЕ хранит — берёт из автора.
- `cities.location` и юзерская точка — PostGIS `geography` + GiST.
- Reverse-geocode и поиск по радиусу — KNN/`ST_DWithin` по индексу.
