# Модерация фото (post-MVP / бета)

> **Статус: спецификация на будущее.** Реализуем **после MVP, на этапе бета-тестирования**.
> В MVP фото проходят без модерации. Здесь — заранее зафиксированная политика и дизайн.

Цель — на загрузке фото профиля проверять две вещи:
1. **Наличие человека/лица** (анти-«пустые»/левые картинки);
2. **NSFW-классификация** (откровенность): явное — сразу reject, спорное — на ручную модерацию.

**Блюра НЕТ** (ни авто, ни на клиенте): фото либо принято и показывается как есть, либо
отклонено. Серая зона решается человеком (accept/reject), а не блюром.

## Классификатор

nsfwjs (или ONNX open_nsfw) даёт классы: `Porn`, `Hentai`, `Sexy`, `Neutral`, `Drawing`.
Важно: модель оценивает **откровенность, а не контекст** — она НЕ отличит «купальник на
пляже» от «бельё в комнате» (обе → `Sexy`). Поэтому контекст автоматикой не различаем:
явный сексуальный контент режем сразу, серую зону — на ручную модерацию.

## Политика (пороги КАЛИБРУЕМЫЕ — вынести в конфиг)

**Любое фото:**
- `Porn`/`Hentai` высоко, либо `Sexy ≥ 0.7` → **REJECT** (сексуальный контент — сразу отклоняем).
- Нет человека/лица → **REJECT**.

**Серая зона → ручная модерация (`PENDING`, без блюра):**
- `0.4 ≤ Sexy < 0.7` → на модерацию (accept/reject). Одинаково для всех фото, включая будущий аватар.

**ALLOW (авто):** `Sexy < 0.4`, не explicit, есть человек.

> Пороги `0.4 / 0.7` — **ориентировочные**, требуют калибровки на реальных данных бета-теста.

## Паттерны контента (для калибровки и модераторов)

**Дозволено (обычно ALLOW):**
- Портрет/селфи, лицо крупным планом, обычная одежда.
- На улице, дома, в кафе; спорт, хобби, активности.
- Пляж/бассейн в купальнике/плавках в умеренном виде.
- (в профиле должен присутствовать сам пользователь; групповые ок, но не вместо себя)

**Серая зона (→ ручная модерация, accept/reject):**
- Откровенный купальник/бельё крупным планом, подчёркнуто провокационные позы (без наготы).
- Глубокое декольте, полупрозрачная одежда, частичное обнажение со спины.
- Пограничный/двусмысленный контент — решает модератор по контексту.

**Недозволено (REJECT):**
- Порнография, половые акты, гениталии, женская грудь / полная нагота.
- Фото без человека (пейзаж/мем/животное/скриншот как основное).
- Чужие фото/знаменитости/водяные знаки, явная реклама/контакты/QR (фрод/спам).
- Насилие, шок-контент, демонстрация оружия.

**Нулевая терпимость — CSAM (сексуальный контент с несовершеннолетними):**
- Мгновенный **reject + блокировка аккаунта**; контент убирается из показа и **сохраняется
  защищённо** (НЕ жёсткое удаление — нельзя уничтожать возможные улики), не уходит в «серую зону».
- **«Эскалация» ≠ автоматическая отправка ПД в органы.** Инцидент эскалируется
  **ответственному лицу/юристу** (не рядовому модератору). Любое раскрытие данных /
  взаимодействие с правоохранителями — **только на законном основании и через юриста**,
  не из кода. Проактивно лить ПД/фото в полицию НЕ делаем.
- Приложение 18+; такого контента быть не должно.

> ⚠️ **Юридическое ревью обязательно.** Весь CSAM-флоу (хранение улик, сроки, кому и как
> сообщать, статус организатора распространения информации, ст. 242.1/242.2 УК, обязанности
> по РФ) согласовать с юристом ДО реализации. Здесь — только продуктовый каркас, не юр-инструкция.

## Влияние на модель `Photo`

- Статус модерации: `moderation` ∈ `APPROVED` | `PENDING` | `REJECTED` (без `BLURRED`).
- **`PENDING`** — на ручной модерации: публично не показывается и не засчитывается, пока
  не станет `APPROVED`. `REJECTED` — не сохраняется/не показывается.
- **2 обязательных фото — только `APPROVED` (не сексуальные).** `complete` требует **≥2
  APPROVED**. Фото сверх двух могут быть `PENDING` — они **не блокируют** создание профиля
  (просто скрыты, пока не одобрены). То есть: 2 чистых обязательны, остальные могут ждать модерацию.
- **Главным можно сделать только `APPROVED` фото** (`PATCH order`); `PENDING`/`REJECTED` — нельзя.
  Модерация **per-photo и постоянна**: уже одобренное фото (в т.ч. прошедшее ручную модерацию —
  напр. купальник) **свапается в главное без повторной проверки**. Отдельного строгого порога
  для аватара нет — гейт один: статус `APPROVED`.

## Архитектура

- Отдельный **сервис модерации** (форк nsfwjs + HTTP API, либо ONNX-модель в Go-воркере).
- Вызов: **синхронно** при `POST /me/profile/photos` (ответ сразу со статусом), либо
  **асинхронно** воркером (фото `PENDING` → `APPROVED`/`REJECTED` по факту).
- Детекция человека/лица — отдельный лёгкий детектор (или тот же сервис).

## Ручная модерация — внутренняя админка (НЕ Telegram)

Канал модерации — **своя админка на своей (РФ) инфраструктуре**, не Telegram:
отправка лиц пользователей в Telegram нарушает локализацию ПД (242-ФЗ),
трансграничную передачу и режим биометрии. Свой инструмент = тот же оператор, данные
не покидают контур.

- **Очередь модерации:** фото в статусе `PENDING` (серая зона / жалобы) →
  модератор видит в админке, жмёт **accept / reject** (без блюра).
- **Веб-пуши модераторам:** при попадании фото в очередь — push-уведомление (**Web Push
  API + service worker + VAPID**, требуется HTTPS). Клик → админка открывается на нужном
  фото. Надёжно на Chrome (десктоп/Android); на iOS — только установленная PWA. Бэк
  хранит push-подписки модераторов и шлёт уведомления.
- **ML как триаж:** авто-reject явного + авто-allow чистого → на людей уходит только серая зона.
- **Жалобы пользователей** падают в ту же очередь — модель на загрузке не идеальна,
  репорты закрывают хвосты.

## Срок и эволюция
- **Бета (первый заход):** ML-триаж (авто reject/allow) + **ручная модерация** серой зоны
  (асинхронно), очередь + веб-пуши.
- **После релиза:** при наличии более сильных мощностей (GPU / крупнее модель) — перейти
  к **моментальной автоматической проверке** (синхронно на загрузке), сокращая/убирая
  ручную модерацию серой зоны и ускоряя UX. Ручная модерация остаётся бэкстопом для жалоб
  и спорного.
